# CLOPE Mushroom Clustering Dashboard

Этот проект представляет собой реализацию алгоритма кластеризации категориальных данных CLOPE на примере классического датасета Agaricus-Lepiota (грибы). 

Отличительная особенность решения — пользовательский интерфейс построен не на стандартном React JSX, а с использованием **MY-UI**: самописной экспериментальной обертки над React, вдохновленной декларативным синтаксисом, стейт-менеджментом и ООП-подходом из Apple SwiftUI.

## 🍄 Особенности реализации задачи

Проект построен с учетом архитектурной методологии Feature-Sliced Design (FSD) и разделен на логические слои:

* **Out-of-Core вычисления (IndexedDB):** Алгоритм (`src/entities/clope/lib/MushroomClusterEngine.ts`) обрабатывает данные порциями, минуя необходимость загружать весь датасет в оперативную память целиком. Данные подгружаются и сохраняются чанками через браузерный IndexedDB (`ClopeStorage`). Это позволяет кластеризовать датасеты неограниченного размера.
* **Web Workers для вычислений:** Кластеризация тысяч записей блокирует главный поток. Для сохранения отзывчивости UI (особенно при взаимодействии с элементами управления) все тяжелые расчеты вынесены в отдельный фоновый поток (`src/features/clope-clustering/worker/clope.worker.ts`).
* **Декларативный UI и Реактивность (`src/shared/my-ui`):** Интерфейс собирается через классы и цепочки вызовов методов. Реализована собственная реактивная система (`MY.State`, `MY.Binding`), которая автоматически точечно обновляет UI при изменении данных без перерендера всего дерева.

## 🏗 Подход к UI: Зачем понадобился MY-UI?

Вместо использования привычных `<div>` и JSX, интерфейс собирается через объектно-ориентированный подход, имитирующий SwiftUI. 

**Ключевые архитектурные решения:**
1. **Строгая типизация API:** Архитектура интенсивно использует возможности TypeScript для обеспечения type-safety: от вывода типов динамических пропсов до работы с дженериками в компонентах-коллекциях (например, `MY.ForEach`).
2. **Использование классических паттернов:** Цепочки визуальных модификаторов (вида `.padding().background()`) реализованы на базе паттерна «Декоратор», где каждый вызов оборачивает базовый класс `MY.View` в новый объект. Реактивная система обновлений построена вокруг паттерна «Наблюдатель» (Observer).
3. **Абстракция низкоуровневой логики:** Громоздкие механизмы — такие как генерация сложных SVG-путей (непрерывное сглаживание углов `MY.RoundedRectangle`) или расчет hit-testing для кнопок — надежно скрыты за лаконичным декларативным интерфейсом.

### Пример синтаксиса (из `MushroomActionButton.tsx`)

Компоненты создаются как классы, наследующие `MY.View`, а их верстка описывается в методе `body()`:

import MY from "@/src/shared/my-ui";

export class MushroomActionButton extends MY.View {
    constructor(
        private readonly text: string,
        private readonly action: () => void,
        private readonly backgroundColor: MY.Color
    ) {
        super();
    }

    body(): MY.View {
        return new MY.Button(
            this.action,
            new MY.Text(this.text)
                .font("headline")
                .padding({ edges: "horizontal", length: 24 })
                .padding({ edges: "vertical", length: 14 })
                .background(this.backgroundColor)
                .foregroundStyle("white")
        )
        .clipShape(new MY.Capsule());
    }
}

## ⚠️ Known Issues & Trade-offs (Ограничения подхода)

Подход с MY-UI применен исключительно в экспериментальных и демонстрационных целях. Для полноценного продакшена архитектура имеет ряд существенных недостатков:

* **GC churn (Нагрузка на Garbage Collector):** При каждом локальном обновлении состояния заново создаются десятки новых экземпляров классов (`new MY.Text()`, `new MY.VStack()` и всех их модификаторов). Хотя кастомный `StateObserver` спасает от полного перерендера дерева, постоянное выделение и очистка памяти (Garbage Collection) под эти объекты может вызывать микрофризы на слабых устройствах.
* **Стилизация через inline-стили:** Модификаторы преобразуются в `React.CSSProperties`. Это блокирует использование медиа-запросов, псевдоклассов (`:hover`, `:focus`) и менее оптимально, чем генерация статического CSS или использование Tailwind.
* **Конфликты Layout-моделей:** Попытка "натянуть" парадигму SwiftUI на веб-стандарты приводит к тому, что поведение контейнеров (например, расчет размеров в `MY.ZStack` или `MY.GeometryReader`) иногда требует неочевидных хаков из-за особенностей CSS Grid и Flexbox.

## 🚀 Быстрый старт

Убедитесь, что у вас установлен Node.js.

1. Установите зависимости:
npm install

2. Запустите сервер разработки:
npm run dev

3. Откройте `http://localhost:3000` в браузере. Вы увидите дашборд:
   * Нажмите «Скачать данные» (произойдет загрузка датасета `agaricus-lepiota.data`, парсинг и сохранение в IndexedDB).
   * Нажмите «Фаза 1» для первичного распределения транзакций.
   * Запустите «Фаза 2» для оптимизации функции глобального критерия (Profit).
